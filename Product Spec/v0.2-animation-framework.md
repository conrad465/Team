# WhiteBoard AI — Animation Framework Spec v0.2

*Last updated: 2026-03-11. Owner: PM skill + human.*

---

## Context & Motivation

Testing rounds on 2026-03-10 revealed a structural problem: the current framework asks the LLM to do precise spatial layout (coordinates, sizing, overlap avoidance) — something LLMs are fundamentally bad at. Scenes that require spatial reasoning (graphs, cycles, overlapping shapes) consistently break. Only linear/sequential topics produce acceptable results.

This spec defines the **animation primitive vocabulary** for a redesigned framework where the LLM describes *what* to show and *when*, and the engine handles *where* and *how*. The animation primitives below are the building blocks the engine uses internally and that scene types compose into coherent visual experiences.

---

## Design Principles

1. **Attention follows motion.** Every animation must serve the narrator — drawing the viewer's eye to the thing being spoken about, exactly when it's spoken about.
2. **One beat, one motion.** Each narration phrase triggers at most one visual event. Never compete for the viewer's attention.
3. **Whiteboard metaphor.** Animations should feel like content being drawn, written, or highlighted on a board — not like a slide deck or video game.
4. **Sensible defaults.** Scene types define default animations for every element role. The LLM should rarely need to override.
5. **Duration from narration.** Animation duration is derived from TTS phrase timing, not specified by the LLM. The engine matches visual pacing to spoken pacing automatically.

---

## Animation Categories (MECE)

Every visual event in a presentation is exactly one of four types. These are mutually exclusive (an event is never two types at once) and collectively exhaustive (there is no visual event that falls outside these four).

| Category | Definition | Element state transition |
|---|---|---|
| **Enter** | Element appears on canvas for the first time | non-existent → visible |
| **Emphasize** | Temporary attention-grab; element returns to its original state afterward | visible → highlighted → visible |
| **Transform** | Element permanently changes one or more properties | visible[A] → visible[B] |
| **Exit** | Element is removed from canvas | visible → non-existent |

## Animation Primitives

### Enter (4 primitives)

These control how a new element first appears. They span two perceptual dimensions: emphasis level (subtle vs. attention-grabbing) and temporal pattern (all-at-once vs. progressive).

| Primitive | Behavior | Best for | Emphasis | Temporal |
|---|---|---|---|---|
| **fade** | Opacity 0→1 | Background elements, labels, supporting content | Low | Simultaneous |
| **pop** | Scale 0→1 with spring overshoot (0→1.15→1) | Key concepts, nodes, primary shapes | High | Simultaneous |
| **draw** | Clip mask reveals element progressively along a direction or path | Lines, arrows, axes, shape outlines, borders | Medium | Progressive |
| **typewrite** | Characters appear one at a time, left to right | Text elements, labels with reading content | Medium | Progressive |

**Selection guide for the engine:**

```
Is it text with >3 words?        → typewrite
Is it a line, arrow, or axis?    → draw
Is it a key concept / focal?     → pop
Otherwise                        → fade
```

### Emphasize (2 primitives)

These draw temporary attention to an element already on canvas. The element always returns to its pre-emphasis state.

| Primitive | Behavior | Best for |
|---|---|---|
| **pulse** | Scale 1→1.2→1 (brief size increase then return) | Revisiting a concept, "look at this again", general-purpose emphasis |
| **flash** | Element's fill color briefly intensifies (brightness +30%) then returns to original | Calling out a specific property, comparison emphasis, "notice the color" |

**Selection guide:** Pulse is the default. Flash is used when the narrator is specifically drawing attention to what an element *represents* (its color, category, or value) rather than its existence.

### Transform (1 primitive)

All permanent property changes use a single animation type.

| Primitive | Behavior | Best for |
|---|---|---|
| **morph** | Smooth interpolation from old property values to new ones over the phrase duration | Any permanent change: color shift, size change, position move, label update |

Morph handles all property types:
- **Numeric** (size, position, opacity, rotation): linear interpolation
- **Color**: interpolate through HSL space for perceptually smooth transitions
- **Text content**: cross-fade (old text fades out, new text fades in simultaneously)

### Exit (2 primitives)

| Primitive | Behavior | Best for |
|---|---|---|
| **fade** | Opacity 1→0 | Default exit — subtle, non-distracting, keeps focus on what remains |
| **shrink** | Scale 1→0, element collapses to center point | Dramatic removal, "this goes away", element is being eliminated or replaced |

**Selection guide:** Fade is the default. Shrink is used when the *removal itself* is the point of the narration (e.g., "costs are eliminated").

## Easing Functions

Easing controls the acceleration profile of an animation. Five options cover the perceptual space:

| Easing | Curve | Default for |
|---|---|---|
| **linear** | Constant speed | draw, typewrite |
| **ease-out** | Fast start → slow finish (decelerating) | fade (enter), morph |
| **ease-in** | Slow start → fast finish (accelerating) | fade (exit), shrink |
| **ease-in-out** | Smooth start and finish | morph (long duration) |
| **spring** | Overshoot then settle | pop, pulse |

The engine selects the appropriate easing automatically based on the animation primitive. LLM override is not expected.

## Scene-Type Default Animations

In the semantic layout pattern framework, each scene type pre-assigns animation primitives to element roles. The LLM only specifies content — the engine animates it.

### Flow

*A → B → C. Nodes connected by directional arrows.*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Node | pop | pulse | fade |
| Arrow / connector | draw | pulse | fade |
| Node label | fade (simultaneous with node) | — | fade |

### Graph

*Axes with curves, data points, and annotations.*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Axis line | draw | — | — |
| Axis label | typewrite | — | — |
| Curve / trend line | draw (along path) | pulse | fade |
| Data point | pop | pulse | fade |
| Annotation | fade | flash | fade |

### Comparison

*Side-by-side panels contrasting two or more items.*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Panel / container | fade | — | fade |
| Item within panel | pop | flash | fade |
| Comparison label | typewrite | — | fade |
| Connector between panels | draw | pulse | fade |

### Cycle

*Circular flow: A → B → C → A.*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Node | pop (sequential around cycle) | pulse | fade |
| Arrow / connector | draw (following cycle direction) | pulse | fade |
| Node label | fade | — | fade |
| Center annotation | typewrite | pulse | fade |

### Timeline

*Events arranged along a temporal axis.*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Timeline axis | draw (left → right) | — | — |
| Event marker | pop | pulse | fade |
| Event label | typewrite | flash | fade |
| Period / span indicator | draw | pulse | fade |

### Hierarchy

*Tree structure: root → branches → leaves.*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Root node | pop | pulse | fade |
| Child node | fade (level by level) | pulse | fade |
| Connector / branch line | draw (parent → child) | — | fade |
| Node label | fade | — | fade |

### Stack

*Layered items (top to bottom or bottom to top).*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Layer | pop (sequential, bottom → top) | pulse | fade |
| Layer label | typewrite | flash | fade |
| Bracket / grouping | draw | — | fade |

### Freeform (fallback)

*Arbitrary element placement — closest to the current system. Used when no structured scene type fits.*

| Element role | Enter | Emphasize | Exit |
|---|---|---|---|
| Shape | pop | pulse | fade |
| Text | typewrite | pulse | fade |
| Line / arrow | draw | pulse | fade |

## Primitive Count Summary

| Category | Count | Primitives |
|---|---|---|
| Enter | 4 | fade, pop, draw, typewrite |
| Emphasize | 2 | pulse, flash |
| Transform | 1 | morph |
| Exit | 2 | fade, shrink |
| **Total** | **9** | |

9 primitives × 5 easings = full animation vocabulary. This is intentionally small. A constrained vocabulary means the LLM has fewer decisions to make (or none, if scene-type defaults apply), and every animation looks polished because each primitive is purpose-built and well-tuned.

## What This Replaces

The current system has 5 animation types (`fade_in`, `pop_in`, `pop_highlight`, `draw_in`, `typewriter`) that conflate category and style. The new system separates these concerns:

| Old | New equivalent |
|---|---|
| `fade_in` | Enter → fade |
| `pop_in` | Enter → pop |
| `pop_highlight` | Emphasize → pulse |
| `draw_in` | Enter → draw |
| `typewriter` | Enter → typewrite |
| *(no equivalent)* | Emphasize → flash |
| *(no equivalent)* | Transform → morph |
| *(no equivalent)* | Exit → shrink |
| *(implicit fade-out on delete)* | Exit → fade |

The old system had no explicit Transform animation — edits snapped to new values or used `pop_highlight`. The new `morph` primitive fills this gap.

## Open Questions

1. **Should morph handle position changes?** If the engine repositions elements when new content enters (e.g., a flow chart auto-spreads), should existing elements morph to their new positions? This would be visually smooth but adds complexity.
2. **Draw direction control.** Should the LLM be able to hint draw direction (left→right, top→bottom, clockwise), or should the engine always infer it from context?
3. **Chained emphasis.** Should there be a way to pulse through a sequence of elements rapidly (e.g., "A, then B, then C all contribute") or is one-at-a-time sufficient?
4. **Audio cues.** Should certain animations (pop, flash) have optional subtle sound effects, or is TTS narration sufficient audio?
